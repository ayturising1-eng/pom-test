AŞAMA 2 - LOGIN RATE-LIMIT TUTARLILIĞI VE SİSTEM YÖNETİCİSİ KİLİT ARACI
======================================================================

KULLANICININ İSTEĞİ
- Yeni kullanıcının birkaç girişten sonra tekrar giriş yapamaması sorununun kanıta dayalı teşhis edilmesi ve düzeltilmesi.
- Başarılı girişlerin rate-limit sayaçlarını temizlediğinin doğrulanması.
- Kullanıcı adı, PIN, PIN-derived Auth şifresi, PLMR_PIN_PEPPER, kullanıcı adı ve IP bucket'larının tutarlılığının kontrol edilmesi.
- Kilit eşiklerinin rastgele değiştirilmemesi.
- Gerekirse yalnızca System Admin için güvenli kilit görüntüleme ve sıfırlama aracı eklenmesi.
- Aşama 3 kapsamına geçilmemesi.

KANITA DAYALI KÖK NEDEN
- Stage 3 SQL içindeki record_pin_login_attempt_v1 başarılı girişte hem username hem mevcut IP bucket'ını siliyor. Bu davranış doğrulandı.
- Kilit eşikleri değiştirilmedi: 5=60 sn, 6=120 sn, 7=300 sn, 8=900 sn, 9=1800 sn, 10=3600 sn, 11=21600 sn, sonrası 86400 sn.
- Asıl doğrulanan tutarlılık hatası şudur: admin_update_user_v1 profiles.username değerini değiştirebiliyor, ancak Supabase Auth şifresi username+PIN+PLMR_PIN_PEPPER üzerinden türetildiği için Auth şifresi otomatik güncellenmiyor. Kullanıcı adı değiştirilen hesaplarda sonraki girişler INVALID_CREDENTIALS üretip rate limit'e girebiliyor.
- Aynı public IP üzerindeki başarısız girişlerin ortak IP bucket'ı nedeniyle farklı kullanıcıları etkileyebileceği doğrulandı. Eşikler güvenlik sebebiyle korunmuştur.
